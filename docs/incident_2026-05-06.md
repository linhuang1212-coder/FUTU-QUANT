# 事故报告: 2026-05-06 持仓被误卖

## 概述

ETF 持仓 (SMH, SLV, GDX) 在一天内被自动卖出三次，造成约 $36 损失。

## 时间线 (美东时间)

| 时间 | 事件 |
|------|------|
| 09:32 | 手动买入 SLV x10, GDX x8, SMH x1 |
| 09:34 | 补仓 SLV x5, GDX x4 |
| 09:36 | **旧 monitor (PID 35584) 误触发止盈，全部卖出** |
| 10:11:02 | 第二次买入 SMH x1, SLV x15, GDX x12 |
| 10:11:08 | **未知程序立即卖出 SMH, GDX，并开始 SLV 疯狂循环** |
| 10:11-11:20 | SLV 被反复 SELL/BUY 数百次（大部分 SELL 失败，少数成功） |
| 11:41 | 发现问题，杀掉所有进程 |
| 11:42 | 第三次买入 SMH x1, SLV x14, GDX x11 |
| 11:42 | 启动修复后的持久化 monitor |

## 根因分析

### 根因1: cost_price=0 误触发 (09:36)
- 旧版 `run_monitor.py` 从富途 API 获取 `cost_price`
- 订单刚成交未结算时，API 返回 `cost_price=0`
- PnL 计算: `(current_price / 0) - 1 = +inf`，被判定为超过 +15% 止盈线
- 三只 ETF 全部被卖出

### 根因2: 多交易程序冲突 (10:11)
- Windows 定时任务 `FUTU-QUANT-AutoStart` 每天 20:25 北京时间自动启动 `start.py --force`
- `start.py` 内含动量轮动 `rebalance()` 逻辑，检测到持仓不符合目标分配
- 与我们手动操作冲突，形成买卖循环
- 大量僵尸 Python 进程（10个+）同时连接富途 API

### 根因3: 无进程隔离机制
- 没有文件锁防止多实例
- 没有检查是否已有交易进程在运行
- 不同脚本对同一账户同时下单

## 修复措施

1. **成本价持久化**: 所有成本价存入 SQLite (`data_store/etf_monitor.db`)，不依赖 API
2. **异常值保护**: cost_price<=0 跳过; PnL>±50% 判为数据异常跳过
3. **文件锁**: `.monitor.lock` 确保只有一个 monitor 实例
4. **禁用定时任务**: `FUTU-QUANT-AutoStart` 已 DISABLE
5. **杀僵尸进程**: 启动前清理所有 Python 进程
6. **Cursor Rule**: `.cursor/rules/trading-safety.mdc` 记录安全规则

## 财务影响

- 初始资金: $2,724
- 最终资金: $2,688 (损失约 $36)
- 损失来源: 多次买卖的价差和手续费
- 当前持仓: SMH x1 @ $544.54, SLV x14 @ $69.91, GDX x11 @ $92.25

## 预防措施

- 系统同一时间只能有一个交易程序运行
- 任何交易脚本启动前必须确认无其他 Python 进程
- 不再使用 Windows 定时任务自动启动交易
- Monitor 使用持久化数据库 + 异常值检查 + 文件锁三重保护
